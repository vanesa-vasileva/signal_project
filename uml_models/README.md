1. Alert Generation System
This diagram models the time decision making pipeline that evaluates incoming patient vitals and routes critical notifications to medical staff. The design separates three responsibilities: rule evaluation, alert creation, and alert dispatch.
AlertGenerator is the baseline. It holds a private list of alert rules and exposes a public method for checking a patient's current readings against those rules. Additional public methods allow rules to be added or removed dynamically at runtime. When a rule is triggered, the generator creates an Alert object and forwards it to the AlertManager. The Alert class is a data container holding a patient identifier, a descriptive message, a timestamp, and a severity classification — reflecting exactly the information clinical staff need to act quickly.
Severity is modelled as an enumeration with four named levels — Low, Medium, High, and Critical providing a typed classification that allows downstream consumers to prioritise notifications based on the severeness of it.
Alert conditions are abstracted through the AlertRule abstract class, which defines a protected vital type field and an abstract evaluation method. Concrete implementations determine when a rule fires. ThresholdRule stores private minimum and maximum values and triggers when a reading falls outside the acceptable range — for instance, a heart rate exceeding 130 beats per minute. RateOfChangeRule stores the previous reading and a maximum permitted change, detecting dangerous sudden spikes or drops in a vital sign. New conditions can be introduced by subclassing AlertRule without modifying existing code, reflecting the open-closed principle.
AlertManager holds a private list of medical staff and dispatches alerts through a send method. The MedicalStaff class receives notifications via a public receive method. The Patient class is intentionally minimal here, exposing only an identifier and a name, since the alert subsystem does not require full clinical records.

2. Data Storage System
This diagram models the subsystem responsible for securely storing, retrieving, and managing the patient vital records. Three concerns are separated into classes: the storage mechanism itself, access control, and audit accountability.
DataStorage serves as the central repository. It holds a private collection of patient records and exposes four public methods: one for writing a new record, two for targeted retrieval by patient identifier or time range, and one that handles outdated entries. Deletion logic is delegated to the RetentionPolicy class, which stores a maximum record age in days and a flag indicating whether old records should be archived rather than deleted. A public evaluation method on this class determines whether any given record qualifies for removal. This separation means that retention rules can be updated without modifying the storage class itself.
PatientData is the atomic unit of storage. It holds a patient identifier, a vital sign type label, a numerical value, and a timestamp. A public accessor for the timestamp supports time range queries. The storage class composes records in a one to many relationship, reflecting that each patient accumulates thousands of entries over time.
DataRetriever is the only sanctioned query interface available to medical staff. It holds a private reference to the storage component and exposes three retrieval methods, each of which requires a MedicalStaff object as a parameter. Before returning any data, it consults the AccessPolicy class, which maps roles to permitted operations and enforces a minimum clearance level. This role based check ensures that, for example, administrative personnel cannot access raw clinical readings.
Every access attempt is recorded by the AuditLog class through a dedicated logging method, supporting regulatory compliance and incident tracing. The MedicalStaff class is kept intentionally simple, storing only a staff identifier, a name, and a role, since no additional detail is required within this subsystem.

3. Patient Identification System
This diagram models the subsystem that ensures every incoming data point is correctly linked to the right hospital patient, and that any failure in that process is detected and handled safely. The design reflects the principle that patient identity is sensitive information — only classes with a genuine need should have access to it.
IdentityManager sits at the top of the hierarchy, serving as the single entry point for the rest of the CHMS. It holds private references to both a patient identifier and a mismatch handler. It exposes two public methods: one performs a full lookup and returns a resolved patient record, while the other returns a boolean signal indicating whether identification succeeded. This explicit failure signal is a deliberate safety choice. 
PatientIdentifier performs the actual matching logic. It queries a PatientRegistry by identifier or by name, and its private match threshold field allows configurable confidence scoring to accommodate minor formatting inconsistencies from the simulator. PatientRegistry stores a private list of patient records and tracks the last time that list was refreshed from the hospital database, ensuring the system always works with up to date information.
HospitalPatient holds the full clinical record, including a patient identifier, a name, a date of birth, and a list of medical history entries — all stored as private fields. Critically, it exposes a public method that produces an AnonymizedPatient object, ensuring that downstream components without a legitimate need for identifiable data receive only a hashed token, a broad age group, and a sex field. This separation embodies the privacy by design principle, preventing unnecessary exposure of sensitive records across subsystem boundaries.
MismatchHandler tracks a list of unmatched identifiers and escalates to administrators after a configurable number of repeated failures, maintaining system integrity without blocking the real time data pipeline.

4. Data Access Layer (Listener Diagram)
This diagram models the subsystem responsible for ingesting raw cardiovascular data from external sources and delivering it in a usable format to the rest of the CHMS. The central design decision is the use of a base DataListener class that defines a shared contract through three public methods — one to start the listener, one to stop it, and one to check whether it is currently running. This abstraction ensures that the rest of the system remains entirely unaware of how data physically arrives.
Three concrete subclasses extend DataListener. The TCPDataListener stores a private host address and port number for socket based streaming. The WebSocketDataListener holds a private endpoint URL and a reconnection interval to handle dropped connections. The FileDataListener stores a private file path and a polling interval. Each subclass overrides the start method with its own protocol specific connection logic.
Once raw data arrives, it is passed to the DataSourceAdapter, which bridges the transport and storage layers. This class holds private references to both a parser and a storage component. Its public methods handle raw input and save the resulting structured record — separating parsing concerns from persistence concerns. The DataParser interface defines a parse method that accepts a raw string and produces a structured patient record, as well as a method to identify the expected format. Two concrete implementations are provided: JsonParser, which uses an internal object mapper, and CsvParser, which relies on a configurable delimiter character. Both produce a PatientData object containing a patient identifier, a vital sign value, a type label, and a timestamp.
The multiplicity annotations in the diagram is as such— one listener to one adapter, one adapter to one parser. This design supports extensibility, as adding a new protocol or data format requires only a new subclass with no changes to core logic.
